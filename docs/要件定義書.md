# 現在地連動型・都バス停留所検索アプリ 要件定義書

作成日：2026年7月18日

---

## 0. 事前調査で確認できたこと（重要）

開発方針を固める前に、構想内の未確定事項（8章・12.1）を調査した。

- **GTFSデータの入手先は実在する。** 東京都交通局は「公共交通オープンデータセンター（ODPT／odpt.org）」を通じて、都バスの **静的データ（GTFS-JP形式）** と **リアルタイムのバスロケーション情報（GTFS-RT形式、JSON）** の両方をCC BY 4.0ライセンスで公開している<cite index="5-1,9-1">。東京都交通局の都バス、都営地下鉄、東京さくらトラムのロケーション情報や時刻表などの情報が公共交通オープンデータセンターより提供されており、都営バスの時刻表等のデータについてはREST APIによるJSON形式のほか、GTFS-JP形式のデータも提供されている</cite>。GTFS-RT形式のバスロケーションデータも2020年から提供開始されている<cite index="6-1">。
  - → **第4段階（バス接近情報の自前表示）は、当初想定より早期に着手できる可能性が高い。** 静的GTFS-JPで停留所・系統マスタを作り、GTFS-RTで接近状況を取得する構成が現実的。

- **公式サイト（tobus.jp）は検索フォーム主体の作りで、固定URLでの直接遷移は保証されない。** 運行情報サービスの停留所検索は部分一致・よみがな検索のフォーム形式であり<cite index="14-1">、スマホ版のURLも `tobus.jp/sp/blsys` 配下の遷移型になっている。停留所IDや系統IDをクエリパラメータとして与えるだけで接近情報画面に直接届くかは、実際にフォーム送信時のリクエストを解析しないと確定できない。
  - → **12.1の課題は解消されていない。** 第2段階着手時に実際の通信をブラウザの開発者ツールで解析する調査タスクが必須。ダメだった場合の代替案（検索画面までの遷移、またはGTFS-RTを使った自前表示への前倒し）を設計に織り込む。

以下の要件定義は、この調査結果を踏まえて構成した。

| 項目 | 内容 |
|---|---|
| プロダクト名（仮） | 都バスナビ（仮称） |
| ゴール | 現在地周辺の都バス停留所・系統・行き先を、停留所名の入力なしに3タップ以内で確認できる |

## 2. 要件定義

### 2.1 機能要件（初期版＝最小実用版のスコープ）

| ID | 要件 | 優先度 |
|---|---|---|
| F-01 | Geolocation APIで現在地（緯度・経度）を取得する | 必須 |
| F-02 | 現在地取得に失敗した場合、手動で地名／駅名を入力して代替検索できる | 必須 |
| F-03 | 停留所マスタ（GTFS由来）から半径200〜300m以内の停留所を検索する | 必須 |
| F-04 | 検索結果を現在地からの距離順（近い順）に並べて表示する | 必須 |
| F-05 | 同一名称でも停留所ID・緯度経度・のりば番号が異なる停留所は別項目として表示する | 必須 |
| F-06 | 各停留所について、通過する系統番号・行き先を一覧表示する | 必須 |
| F-07 | 系統・行き先をタップすると都バス公式サイトの該当ページ（またはそれに最も近いページ）を開く | 必須 |
| F-08 | 近隣停留所が1件、または系統が1件しかない場合は選択画面を自動スキップする | 必須 |
| F-09 | 現在時刻に運行していない系統（始発前・終電後など）は候補から除外、または区別表示する | 推奨（第2段階） |
| F-10 | よく使う系統・最近使った停留所を記憶し優先表示する | 任意（第3段階） |
| F-11 | 地図上から停留所を選択できる | 任意（第3段階） |
| F-12 | アプリ内でバス接近情報（あと何停留所・何分）を自前表示する | 任意（第4段階） |

### 2.2 非機能要件

| 分類 | 要件 |
|---|---|
| 性能 | 起動から近隣停留所一覧表示まで3秒以内（4G回線相当） |
| 性能 | 停留所マスタJSONは対象エリア分割等で1回の読み込みを数百KB以下に抑える |
| 操作性 | 起動〜公式ページ到達までのタップ数を3回以内に収める |
| 操作性 | ボタンは片手操作を想定し、タップ領域は44px角以上を確保する |
| 可用性 | 通信が不安定でも、一度読み込んだ停留所データはオフラインでも参照できる（Service Worker） |
| 保守性 | GTFSデータの入れ替えのみで停留所・系統情報を更新できる構成にする |
| 保守性 | 位置情報処理・停留所検索ロジック・公式サイト遷移ロジックをモジュールとして分離する |
| プライバシー | 位置情報は端末内処理のみで、サーバーへの送信・保存を行わない（第1〜3段階は完全クライアントサイド） |
| ライセンス | GTFS-JP／GTFS-RTデータはCC BY 4.0のためクレジット表記を画面内（フッター等）に明記する |

### 2.3 スコープ外（初期版で扱わないこと）

- ネイティブアプリ化（Android／iOS）
- 経路検索（乗り換え案内）
- プッシュ通知・ウィジェット
- ユーザーアカウント・サーバー側でのデータ保存

## 3. システム構成

```
[ブラウザ / PWA]
  ├─ UI層（HTML/CSS/JS、将来的にReact等へ拡張可）
  ├─ 位置情報モジュール（Geolocation API）
  ├─ 停留所検索モジュール（距離計算・絞り込み）
  ├─ 公式サイト遷移モジュール（URL組み立て or 検索画面誘導）
  └─ Service Worker（オフラインキャッシュ、PWA化）

[静的データ（ビルド時生成）]
  ├─ stops.json      … 停留所マスタ（停留所ID・名称・緯度経度・のりば情報）
  ├─ routes.json     … 系統マスタ（系統番号・行き先・運行会社）
  └─ stop_routes.json … 停留所×系統×方向の対応表

[データ加工パイプライン（開発時のみ／オフライン処理）]
  GTFS-JP CSV一式（ODPT提供）
     → Pythonスクリプトで変換・正規化
     → 上記JSON群を生成
     → 静的ホスティング（GitHub Pages / Cloudflare Pages 等）へ配置
```

## 4. データ設計

### 4.1 GTFS-JPから抽出・加工する項目

| GTFSファイル | 抽出項目 | 用途 |
|---|---|---|
| stops.txt | stop_id, stop_name, stop_lat, stop_lon, platform_code | 停留所検索・のりば区別 |
| routes.txt | route_id, route_short_name（系統番号）, agency_id | 系統表示 |
| trips.txt | trip_id, route_id, trip_headsign（行き先） | 行き先表示 |
| stop_times.txt | trip_id, stop_id, stop_sequence | 停留所ごとの通過系統特定 |
| calendar.txt / calendar_dates.txt | service_id, 曜日・運行日 | 現在時刻運行判定（F-09） |

### 4.2 加工後のstop_routes.jsonイメージ

```json
{
  "stop_id": "12345",
  "stop_name": "○○駅前",
  "platform_code": "1番のりば",
  "lat": 35.6895,
  "lon": 139.6917,
  "routes": [
    { "route_id": "to01", "route_name": "都01", "headsign": "渋谷駅前行き" },
    { "route_id": "to02", "route_name": "都02", "headsign": "大塚駅前行き" }
  ]
}
```

### 4.3 GTFS-RT（第4段階用）

バスロケーション情報はJSON形式のリアルタイムAPIとしてODPTから取得可能<cite index="19-1">。第4段階では `trip_id` を軸に静的データとリアルタイムデータを突合し、「あと何停留所前か」を自前計算する設計とする。

## 5. 開発ワークフロー（フェーズ別）

### フェーズ0：調査・環境準備（着手前に必須）
- [ ] ODPT（公共交通オープンデータセンター）のアカウント登録・利用規約確認
- [ ] 都バスGTFS-JP／GTFS-RTデータの取得・データ構造の実機確認
- [ ] tobus.jp（都バス運行情報サービス）の検索〜結果表示までの通信をブラウザ開発者ツールで解析し、直接遷移可能なURLパラメータが存在するか検証
- [ ] 開発・公開環境の決定（GitHub Pages を第一候補とする）

### フェーズ1：データパイプライン構築
- [ ] Pythonでstops.txt / routes.txt / trips.txt / stop_times.txtを読み込み、停留所×系統×行き先の対応表を生成するスクリプトを作成
- [ ] 対象エリア（まず自宅・職場周辺など狭域）に絞ったテストデータで動作確認
- [ ] 生成したJSONの容量・構造を検証

### フェーズ2：最小実用版（MVP）実装
- [ ] Geolocation APIで現在地取得
- [ ] Haversine公式等による距離計算・半径絞り込みロジック実装
- [ ] 距離順一覧表示のUI実装（停留所名・距離・のりば・系統・行き先）
- [ ] タップで公式サイトへ遷移する処理実装（フェーズ0の調査結果に応じて「直接URL」または「検索画面誘導」を選択）
- [ ] 実機（Android Chrome）での動作確認

### フェーズ3：操作性改善
- [ ] 選択肢が1件のときの自動スキップ（F-08）
- [ ] 現在時刻による運行系統の絞り込み（F-09）
- [ ] localStorageでの「よく使う系統」「最近使った停留所」記憶（F-10）
- [ ] PWA化（manifest.json・Service Worker追加、オフラインキャッシュ）
- [ ] 地図UI（Leaflet等）からの停留所選択（F-11）

### フェーズ4：バス接近情報の自前表示
- [ ] GTFS-RTのポーリング取得
- [ ] trip_idベースでの現在位置・遅延突合ロジック
- [ ] 「あと何停留所」「推定到着時刻」表示UI
- [ ] 運休・迂回情報の表示

### フェーズ5：評価・展開判断
- [ ] 初期版の開発指標（10章）達成状況の確認
- [ ] Androidネイティブアプリ化の要否判断（位置情報連携・通知・ウィジェット等が必要になった時点で検討）

## 7. リスクと対策一覧

| リスク | 内容 | 対策 |
|---|---|---|
| 公式サイト直接遷移の不可 | tobus.jpがセッション依存の可能性 | フェーズ0で通信解析。ダメなら検索画面誘導＋入力項目自動設定、または第4段階を前倒しして自前表示に切り替え |
| 同名停留所の誤認識 | 名称だけでは道路の反対側・複数のりばを区別できない | stop_id・緯度経度・platform_codeを組み合わせて一意に識別（4.1参照） |
| GPS誤差 | 道路反対側の停留所が最上位に出る可能性 | 初期版は距離順表示に留め、将来的に進行方向・移動方向を加味 |
| GTFSデータの更新遅延 | 路線改廃にデータ反映が追いつかない可能性 | データ加工パイプラインをスクリプト化し、GTFS差し替えのみで再ビルドできる構成にする |
| ライセンス遵守漏れ | CC BY 4.0のクレジット表記漏れ | アプリのフッター・Aboutページにデータ提供元を明記 |

## 8. 初期版（MVP）の完了基準

- Android実機でGeolocation APIから現在地を取得できる
- 半径200〜300m以内の停留所を距離順に表示できる
- 同名・複数のりばの停留所を区別して表示できる
- 各停留所の系統・行き先を表示できる
- タップから3回以内の操作で都バス公式サイトの関連ページに到達できる
- オフライン状態でも直前に読み込んだ停留所一覧は表示できる

## 9. 次のアクション（すぐ着手できること）

1. ODPT（odpt.org／公共交通オープンデータセンター）でアカウント登録し、都バスGTFS-JPデータを実際にダウンロードする

3. 上記2つの結果をもとに、フェーズ2（MVP実装）の設計を確定する
